## Question 1: OTel Collector → Grafana Client Cert (Cat 9 Gap)

**Question**: Category 9 (`{otel-collector-contrib,grafana-otel-lgtm}-https-client-entity-{PS-ID}-{sqlite,postgres}-{1,2}/` + `admin`) generates per-PS-ID app instance client certs and one `admin` cert for Grafana LGTM and OTel Collector. These certs authenticate *to* those services. However, OTel Collector Contrib itself needs a *client cert* when forwarding telemetry *to* Grafana LGTM's OTLP ingest (:14317/:14318). That client-as-OTel role is not present in any category. Which resolution?

**Context**: Cat 9 name pattern: `{grafana-otel-lgtm,otel-collector-contrib}-https-client-entity-{admin}/{PS-ID}-{sqlite,postgres}-{1,2}`. The `admin` cert in Cat 9 is for human operator UI access, not for OTel-to-Grafana service-to-service forwarding. The Grafana LGTM `grafana-otel-lgtm-https-client-issuing-ca/` is used to verify incoming client certs from both app instances AND OTel Collector — but OTel Collector has no cert issued by that CA yet.

**A)** Reuse the existing `grafana-otel-lgtm-https-client-entity-admin/` cert for OTel Collector's outbound connection to Grafana OTLP ingest. The `admin` cert is already in Cat 9. This is a semantic stretch (admin cert used for service-to-service), but avoids adding a new cert.

**B)** Add `grafana-otel-lgtm-https-client-entity-otel-collector-contrib/` to Cat 9 in `tls-structure.md` and add it to the generator in a v11 follow-on patch. This is the cleanest semantic separation.

**C)** Skip mTLS for OTel→Grafana forwarding entirely (server TLS only, no client cert). Grafana LGTM's OTLP ingest (:14317/:14318) only verifies the server cert; OTel Collector does not present a client cert. Reduces security but avoids category change.

**D)** Document the gap in tls-structure.md as a known limitation and defer to a future framework-v13.

**E)**

**Answer**:

**Rationale**: This is the most architecturally significant gap found in the deep analysis. The answer determines whether Phase 5 requires a v11 generator patch (B), a semantic workaround (A), or a security reduction (C/D).

## Question 2: Cat 14 SQLite App PostgreSQL Client Certs

**Question**: Category 14 generates PostgreSQL client certs for ALL 4 app instance suffixes: `{sqlite-1, sqlite-2, postgres-1, postgres-2}`. The sqlite-1 and sqlite-2 instances use in-memory SQLite and NEVER connect to PostgreSQL. Their Cat 14 certs are generated but unused. Which resolution?

**Context**: Current tls-structure.md defines Cat 14 as: `postgres-tls-client-entity-{leader,follower}-{PS-ID}-{sqlite,postgres}-{1,2}/`. Total per PS-ID: 8 directories (2 DB roles × 4 app instances). The sqlite-1 and sqlite-2 variants would consume 4 of these 8 directories unnecessarily.

**A)** Keep generating for all 4 suffixes (`sqlite-1`, `sqlite-2`, `postgres-1`, `postgres-2`). The extra certs are harmless, code is simpler, and a future sqlite-with-external-pg use case might need them.

**B)** Change Cat 14 to only generate for `postgres-1` and `postgres-2`. Update tls-structure.md Cat 14 description and the generator. Reduces directory count from 8 to 4 per PS-ID (from 80 to 40 at suite level).

**C)** Keep all 4 suffixes in tls-structure.md documentation but only mount postgres-1/postgres-2 certs in compose volume binds. Let the unnecessary certs exist on disk but never mount them into sqlite containers.

**D)**

**E)**

**Answer**:

**Rationale**: Affects both tls-structure.md accuracy and generator code in v11 follow-on. Option A is pragmatic; Option B is correct. Option C is a documentation/operational split.

## Question 3: Cross-Compose Certificate Volume Sharing Strategy

**Question**: `pki-init` runs in a PS-ID compose file (e.g. `skeleton-template/compose.yml`), populates `skeleton-template-certs` named volume. `shared-postgres/compose.yml` and `shared-telemetry/compose.yml` are included via `include:` — they don't define their own `pki-init` and cannot reference a volume defined in the PS-ID compose. Which volume sharing approach should shared-postgres and shared-telemetry use to access certs generated by pki-init?

**Context**: Current `skeleton-template/compose.yml` mounts `skeleton-template-certs:/certs:ro` in app instances. The `shared-postgres` and `shared-telemetry` composes are included *into* skeleton-template's compose via `include:`. Docker Compose merges all included files into one project. Named volumes defined in the PS-ID compose file ARE visible to included files within the same compose project. The volume name `skeleton-template-certs` is PS-ID-specific (from a template with `__PS_ID__` placeholder).

**A)** Include-merged approach: shared-postgres and shared-telemetry reference the cert volume by its template name (e.g., `__PS_ID__-certs`). When included into a PS-ID compose, the `__PS_ID__` placeholder is substituted at template-generation time. This volume is defined in the PS-ID compose and visible to all included services in the same project.

**B)** External named volume: define a separate `cryptoutil-shared-certs` volume name (constant, not parameterized) that both the PS-ID compose and shared composes mount from. The PS-ID's pki-init writes to this volume; shared-postgres and shared-telemetry read from it. Requires all three compose files to declare `cryptoutil-shared-certs` consistently.

**C)** Bind mount from a host directory (e.g., `${CERTS_DIR:-./certs}:/certs:ro`). Set `CERTS_DIR` environment variable or use compose `env_file`. Works but requires host path coordination across multiple compose projects.

**D)** Each shared stack runs its own `pki-init` scoped to the shared tier (postgres-only or telemetry-only domain). This contradicts v11 Decision D6 (no pki-init in shared-telemetry) and would require a new tier domain.

**E)**

**Answer**:

**Rationale**: This is the most operationally critical decision — it determines whether shared-postgres and shared-telemetry can even access certs. Option A is cleanest (leverages Docker Compose include/merge semantics already in use). Option B requires a global constant volume name. The answer must be confirmed before writing any Phase 1/3/5/6 compose changes.

## Question 4: Grafana OTLP Ingest TLS Capability

**Question**: `grafana/otel-lgtm` is an all-in-one image running Grafana + Loki + Tempo + Mimir + embedded OTel Collector. The embedded OTel Collector (inside the image) receives OTLP on :4317/:4318 and forwards to internal Loki/Tempo/Mimir. External OTel Collector Contrib connects to `:14317/:14318` (host-mapped). Does `grafana/otel-lgtm` support mTLS on its embedded OTLP receiver (:14317/:14318)?

**Context**: No official documentation found on mTLS support for grafana/otel-lgtm OTLP ingest ports. The Grafana HTTPS UI config (Q1=B, D1) uses `grafana.ini` for `[server]` settings. The embedded OTel Collector inside grafana/otel-lgtm is configured differently — via an internal config that may not be user-accessible. The external `otel-collector-contrib` we control fully supports mTLS.

**A)** Assume grafana/otel-lgtm supports mTLS on OTLP ingest via env vars (`OTEL_COLLECTOR_CONFIG_FILE` or similar) — implement Phase 5 mTLS and test at runtime. If it fails, fall back to Option C.

**B)** Prototype first: before Phase 5, spin up `grafana/otel-lgtm` locally, attempt to configure OTLP mTLS, and confirm it works before committing the approach in the plan.

**C)** Use server-TLS-only for OTel→Grafana forwarding (not mTLS). OTel Collector verifies Grafana's server cert (via Cat 1 truststore), but Grafana does not verify OTel's client cert. This is still encrypted, just not mutually authenticated.

**D)** Replace `grafana/otel-lgtm` with separately deployed Grafana + Loki + Tempo + Mimir containers (each fully configurable). This gives full mTLS control but significantly increases infrastructure complexity.

**E)**

**Answer**:

**Rationale**: Phase 5 design depends entirely on this answer. Option A proceeds optimistically; Option B defers Phase 5 start pending a prototype spike; Option C accepts reduced security for operational simplicity; Option D is a major architecture change. This must be answered before Phase 5 tasks are finalized.
