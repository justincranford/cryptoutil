# TLS Structure

This document defines the required `/certs` volume layout generated by `pki-init`.

Key parameterization aspects include:
- PS-ID level
- PRODUCT level
- SUITE level
- Shared PostgreSQL/OTel/Grafana

## Naming Model

Canonical PS-ID app instance names:
- `{PS-ID}-app-sqlite-1`
- `{PS-ID}-app-sqlite-2`
- `{PS-ID}-app-postgres-1`
- `{PS-ID}-app-postgres-2`

Canonical tier IDs:
- SUITE: `cryptoutil`
- PRODUCT: `sm`, `jose`, `pki`, `identity`, `skeleton`
- PS-ID: `sm-kms`, `sm-im`, `jose-ja`, `pki-ca`, `identity-authz`, `identity-idp`, `identity-rs`, `identity-rp`, `identity-spa`, `skeleton-template`

## PKI INIT

`pki-init` takes two positional parameters:
1. **PKI-INIT-DOMAIN** (`<tier-id>`) — canonical tier ID; one of the 16 valid domains below
2. **TARGET-DIRECTORY** (`<target-dir>`) — target directory root; e.g. `/certs`, `/tmp`

All output files are created under `TARGET-DIRECTORY/PKI-INIT-DOMAIN/`. If that subdirectory exists and is not empty, `pki-init` refuses to generate any files. If the subdirectory does not exist, or exists but is empty, `pki-init` creates the full directory tree and generates all required files.

### Valid PKI-INIT-DOMAIN Values (16 total)

| Tier | Domain |
|------|--------|
| Suite | `cryptoutil` |
| Product | `sm`, `jose`, `pki`, `identity`, `skeleton` |
| PS-ID | `sm-kms`, `sm-im`, `jose-ja`, `pki-ca`, `identity-authz`, `identity-idp`, `identity-rs`, `identity-rp`, `identity-spa`, `skeleton-template` |

Examples:
1. `pki-init cryptoutil        /certs`   → output: `/certs/cryptoutil/` (suite scope, 666 dirs)
2. `pki-init sm                /certs`   → output: `/certs/sm/` (product scope, 150 dirs for 2 PS-IDs)
3. `pki-init identity          /certs`   → output: `/certs/identity/` (product scope, 4 PS-IDs)
4. `pki-init sm-kms            /tmp`     → output: `/tmp/sm-kms/` (PS-ID scope, 90 dirs)
5. `pki-init skeleton-template /tmp`     → output: `/tmp/skeleton-template/` (PS-ID scope, 90 dirs)

## File Format Convention

Each directory name IS the base name for the files it contains. Two directory types exist:

| Type | Files | Description |
|------|-------|-------------|
| **keystore** | `SAME-AS-KEYSTORE-DIR-NAME.p12`, `SAME-AS-KEYSTORE-DIR-NAME.crt`, `SAME-AS-KEYSTORE-DIR-NAME.key` | PKCS#12 bundle + PEM cert + PEM private key |
| **truststore** | `truststore/SAME-AS-KEYSTORE-DIR-NAME.p12`, `truststore/SAME-AS-KEYSTORE-DIR-NAME.crt` | PKCS#12 bundle + PEM cert (NO private key); in `truststore/` subdir of keystore dir |

**Rule**: Any combination of `{truststore}` and `{key}` is ignored — a truststore NEVER contains a private key.

**Rule**: `pki-init` generates truststores **only for CA certificates** as a `truststore/` subdirectory within the keystore directory. End-entity (leaf) certificates NEVER receive a `truststore/` subdirectory — trust is established via the issuing CA's truststore. `pki-init` never generates self-signed leaf certificates.

## Requirements

Goal: generate a complete set of certs for the deployment of the SUITE, a PRODUCT, or a PS-ID.

For every PS-ID, there are 4 app instances:
1. `{PS-ID}-app-sqlite-1`
2. `{PS-ID}-app-sqlite-2`
3. `{PS-ID}-app-postgres-1`
4. `{PS-ID}-app-postgres-2`

Required domains (16 categories):

**1. Public Global HTTPS Server CAs:**
- One globally shared CA chain (root + issuing) for all public HTTPS server leaf certs in the deployment (app instances, Grafana LGTM, OTel Collector).
- Both keystore and truststore directories for root and issuing.
- Total: 4 directories (2 CA tiers × 2 store types); same across PS-ID, PRODUCT, and SUITE levels.

**2. Public Grafana LGTM + OTel Collector Server Certs:**
- One server leaf cert for Grafana LGTM HTTPS UI (port 3000), issued by the global server CA.
- One server leaf cert for OTel Collector Contrib HTTPS (gRPC :4317, HTTP :4318 share the same cert), issued by the global server CA.
- Keystore only per service (leaf cert — trust established via issuing CA's truststore).
- Total: 2 directories (2 services × 1 store type); same across PS-ID, PRODUCT, and SUITE levels.

**3. Public PS-ID App Server Certs:**
- One server leaf cert per app instance (4 per PS-ID: sqlite-1, sqlite-2, postgres-1, postgres-2), issued by the global server CA.
- Keystore only per instance (leaf cert — trust established via issuing CA's truststore).
- Total per PS-ID: 4 directories (4 instances × 1 store type).
- Total per PRODUCT: 4 × (PS-ID count in product).
- Total per SUITE: 40 directories (4 instances × 10 PS-IDs × 1 store type).

**4. Public PS-ID HTTPS Client CAs:**
- Per-instance CA chain (root + issuing) for client authentication on each app instance.
- {sqlite-1,sqlite-2,postgres} = 3 PKI domains per PS-ID; each has root and issuing CAs; both keystore and truststore.
- Total per PS-ID: 12 directories (3 PKI domains × 2 CA tiers × 2 store types).
- Total per PRODUCT: 12 × (PS-ID count in product).
- Total per SUITE: 120 directories.

**5. Public PS-ID HTTPS Client Certs:**
- Client leaf certs per API path prefix (`browseruser`, `serviceuser`) and per realm type, grouped into 3 PKI domains: `sqlite-1`, `sqlite-2`, `postgres` (postgres-1 and postgres-2 share client identity per Decision 7).
- Leaf certs per PKI domain: `{browseruser,serviceuser}` × `{realm}` — realm values are dynamic per PS-ID, read from `registry.yaml` at pki-init runtime (Decision 8).
- Keystore only per leaf cert (leaf cert — trust established via issuing CA's truststore).
- Total per PS-ID: `2 × |realms| × 3` directories. Examples assume 2 realms (`file`, `db`): 12 directories.
- Total per PRODUCT: `2 × |realms| × 3 × (PS-ID count in product)`.
- Total per SUITE: `2 × |realms| × 3 × 10`; assuming 2 realms: 120 directories.

**6. Private PS-ID HTTPS Mutual TLS CAs (Admin Channel):**
- Per-instance CA chain (root + issuing) for the private admin HTTPS channel; mutual TLS required.
- {sqlite,postgres} × {1,2} = 4 instances per PS-ID; each has root and issuing CAs; both keystore and truststore.
- Total per PS-ID: 16 directories (4 instances × 2 CA tiers × 2 store types).
- Total per PRODUCT: 16 × (PS-ID count in product).
- Total per SUITE: 160 directories.

**7. Private PS-ID HTTPS Mutual TLS Leaf Certs (Admin Channel):**
- One mTLS leaf cert per app instance, with both Client Auth and Server Auth EKU; for use by the app instance's private admin HTTPS connections.
- `{mutual}` is a literal keyword indicating the combined client+server role.
- Keystore only per instance (leaf cert — trust established via issuing CA's truststore).
- Total per PS-ID: 4 directories (4 instances × 1 store type).
- Total per PRODUCT: 4 × (PS-ID count in product).
- Total per SUITE: 40 directories.

**8. Public Grafana LGTM + OTel Collector Client CAs:**
- Per-service CA chain (root + issuing) for client authentication toward Grafana LGTM and OTel Collector.
- Both keystore and truststore per CA tier per service.
- Total: 8 directories (2 services × 2 CA tiers × 2 store types); same across PS-ID, PRODUCT, and SUITE levels.

**9. Public Grafana LGTM + OTel Collector Client Certs:**
- Client leaf certs for each PS-ID app instance, one `admin` cert (human operator UI/API access), and one `infra` cert (OTel Collector → Grafana LGTM OTLP ingest service-to-service), issued by the respective client CA.
- `{PS-ID}-{sqlite,postgres}-{1,2}` expands to all 4 app instances per PS-ID; `admin` is one shared cert for operators; `infra` is one shared cert for service-to-service OTel→Grafana forwarding.
- Keystore only per leaf cert per service (leaf cert — trust established via issuing CA's truststore).
- Total per PS-ID: 12 directories (2 services × (4 per-PS-ID instances + 1 admin + 1 infra) × 1 store type).
- Total per PRODUCT: 2 services × (4×N per-PS-ID instances + 1 admin + 1 infra) × 1 store type = 4+8×N.
- Total per SUITE: 2 services × (4×10 per-PS-ID instances + 1 admin + 1 infra) × 1 store type = 84 directories.

**10. Public PostgreSQL Server CAs:**
- One globally shared CA chain (root + issuing) for both the leader and follower PostgreSQL server instances.
- Both keystore and truststore per CA tier.
- Total: 4 directories (2 CA tiers × 2 store types); same across PS-ID, PRODUCT, and SUITE levels.

**11. Public PostgreSQL Server Certs:**
- One server leaf cert for the leader DB instance; one for the follower DB instance.
- Keystore only per instance (leaf cert — trust established via issuing CA's truststore).
- Total: 2 directories (2 instances × 1 store type); same across PS-ID, PRODUCT, and SUITE levels.

**12. Public PostgreSQL Client CAs:**
- One globally shared CA chain (root + issuing) for all clients connecting to PostgreSQL (covering both leader and follower).
- Both keystore and truststore per CA tier.
- Total: 4 directories (2 CA tiers × 2 store types); same across PS-ID, PRODUCT, and SUITE levels.

**13. Public PostgreSQL Replication Client Certs:**
- One client leaf cert for leader↔follower replication (leader cert, follower cert).
- Keystore only per instance (leaf cert — trust established via issuing CA's truststore).
- Total: 2 directories (2 instances × 1 store type); same across PS-ID, PRODUCT, and SUITE levels.

**14. Public PS-ID PostgreSQL App Client Certs:**
- Per-PS-ID client leaf certs for postgres-1 and postgres-2 app instances only (sqlite-1 and sqlite-2 instances use in-memory SQLite and never connect to PostgreSQL).
- One cert per postgres instance per DB role (leader, follower).
- Keystore only per leaf cert (leaf cert — trust established via issuing CA's truststore).
- Total per PS-ID: 4 directories (2 DB roles × 2 postgres instances × 1 store type).
- Total per PRODUCT: 4 × (PS-ID count in product).
- Total per SUITE: 40 directories.

Required logical layout:

```text
TARGET-DIRECTORY/{PKI-INIT-DOMAIN}/
  public-https-server-{root,issuing}-ca{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}
  public-https-server-{entity}-{PS-ID}-{sqlite,postgres}-{1,2}{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}
  public-https-server-{entity}-{otel-collector-contrib,grafana-otel-lgtm}{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}

  public-https-client-{root,issuing}-ca-{PS-ID}-{sqlite-1,sqlite-2,postgres}{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}
  public-https-client-{entity}-{PS-ID}-{sqlite-1,sqlite-2,postgres}-{browseruser,serviceuser}-{realm}{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}

  private-https-mutual-{root,issuing}-ca-{PS-ID}-{sqlite,postgres}-{1,2}{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}

  {otel-collector-contrib,grafana-otel-lgtm}-https-client-{root,issuing}-ca{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}
  {otel-collector-contrib,grafana-otel-lgtm}-https-client-{entity}-{PS-ID}-{sqlite,postgres}-{1,2}{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}
  {otel-collector-contrib,grafana-otel-lgtm}-https-client-{entity}-{admin,infra}{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}

  postgres-tls-server-{root,issuing}-ca{/,/truststore/}SAME-AS-KEYSTORE-DIR-NAME.{p12,crt,key}

```

## Policy Alignment

- **Private admin channel** (`private-https-mutual-{PS-ID}-*`): Mutual TLS required; both server and client auth use the same combined leaf cert per instance (`private-https-mutual-entity-{PS-ID}-*`).
- **Public HTTPS server** (`public-https-server-entity-{PS-ID}-*`): Server cert issued by global server CA; client TLS authentication optional.
- **Public HTTPS client** (`public-https-client-entity-{PS-ID}-{sqlite-1,sqlite-2,postgres}-{browseruser,serviceuser}-*`): Client certificates issued per PKI domain and per API path prefix (`/browser/` → `browseruser`, `/service/` → `serviceuser`) and per realm type, per PS-ID.
- **PostgreSQL connections** (`postgres-tls-{server,client}-*`): Mutual TLS and username+password required; mTLS required for all app-instance-to-DB and leader↔follower replication connections. Only postgres-1 and postgres-2 app instances connect to PostgreSQL; sqlite-1 and sqlite-2 do not.
- **OTel Collector OTLP** (`public-https-server-entity-otel-collector-contrib` + `otel-collector-contrib-https-client-*`): Server cert for :4317/:4318; client certs per PS-ID per app instance, one `admin` cert for operators, and one `infra` cert for OTel Collector’s outbound connection to Grafana LGTM OTLP ingest.
- **Grafana LGTM OTLP ingest** (`grafana-otel-lgtm-https-client-*`): Client certs per PS-ID per app instance, one `admin` cert for UI access, and one `infra` cert for service-to-service OTel→Grafana forwarding; server cert issued by global server CA.
- **Grafana LGTM HTTPS UI** (port 3000): Server cert in `public-https-server-entity-grafana-otel-lgtm/`, issued by global server CA.

## Example: sm (PRODUCT)

Command: `pki-init sm /tmp`

PS-IDs in product: `sm-kms`, `sm-im`. Realm values assumed: `file`, `db`.

Global directories (categories 1, 2, 8, 10-13) are identical to the skeleton-template example. Only PS-ID-specific and product-scoped directories differ.

```text
/tmp/sm/

  # Category 1: Global HTTPS Server CAs (4 dirs = 2 keystore + 2 truststore subdirs) — same as PS-ID level
  public-https-server-root-ca/
  public-https-server-root-ca/truststore/
  public-https-server-issuing-ca/
  public-https-server-issuing-ca/truststore/

  # Category 2: Grafana/OTel Server Certs (2 dirs) — same as PS-ID level
```
