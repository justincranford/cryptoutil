// TLSProvisionMode controls how server certificate material is sourced at server startup.
// This enum is about server certificate provisioning, not client certificate authn policy.
type TLSProvisionMode string

const (
    // Static: do not generate; use pre-provided cert chain and key.
    TLSProvisionModeStatic TLSProvisionMode = "static"
    // Mixed: use provided CA cert chain and key, use it to generate and issue leaf cert chain and key.
    TLSProvisionModeMixed TLSProvisionMode = "mixed"
    // Auto: generate CA cert chain and key, and use it to generate and issue leaf cert and key.
    TLSProvisionModeAuto TLSProvisionMode = "auto"
)

```go
// TLSClientPolicy controls TLS-layer client certificate request/require/verification behavior.
// This enum is about client certificate auth policy, not certificate provisioning.
// Ordered weakest -> strongest (overall): Off, Ask, Require, Verify, Enforce.
type TLSClientPolicy string

const (
    // Off (Go TLS: NoClientCert): do not request a client cert.
    // PoP: no | authn: no | ident: no | authz: no.
    TLSClientPolicyOff TLSClientPolicy = "off"
    // Ask (Go TLS: RequestClientCert): request a client cert, but do not require one and do not verify X.509 chain/CA trust.
    // PoP: conditional (yes only if client sends cert) | authn: conditional | ident: conditional/weak | authz: no.
    TLSClientPolicyAsk TLSClientPolicy = "ask"
    // Require (Go TLS: RequireAnyClientCert): require a client cert, but do not verify X.509 chain/CA trust.
    // PoP: yes | authn: yes (key possession only) | ident: weak/untrusted | authz: no.
    TLSClientPolicyRequire TLSClientPolicy = "require"
    // Verify (Go TLS: VerifyClientCertIfGiven): request a client cert, do not require one, and if sent verify X.509 chain/CA trust, validity window, and key usage rules.
    // PoP: conditional (yes only if client sends cert) | authn: conditional/strong-when-present | ident: conditional/trusted-when-present | authz: no.
    TLSClientPolicyVerify TLSClientPolicy = "verify"
    // Enforce (Go TLS: RequireAndVerifyClientCert): require a client cert and verify X.509 chain/CA trust, validity window, and key usage rules.
    // PoP: yes | authn: yes | ident: yes | authz: no.
    TLSClientPolicyEnforce TLSClientPolicy = "enforce"
)
```

Security note for review: Ask and Require are weaker than Verify and Enforce because TLS does not perform CA trust-path validation in those two modes.
TLS-layer checks here mean X.509 processing during handshake (chain build to trusted roots, validity period, and key-usage constraints). They do not prove application identity/authorization by themselves.
Application-layer compensating controls (especially for Ask/Require): map certificate subject/SAN to an allowlisted service identity, require tenant/realm-bound authz checks, enforce audience/scope ACLs, and record mTLS identity in audit logs before privileged operations.
